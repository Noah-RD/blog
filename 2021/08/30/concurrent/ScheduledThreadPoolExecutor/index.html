ScheduledThreadPoolExecutor 源码学习

导读

调度线程池是线程池的一个特定版本实现，用于执行定时任务。其继承自 ThreadPoolExecutor。

上图左侧是线程池继承图，后侧是 Task 继承图。

调度线程池的核心逻辑是，将提交的可执行任务(Runnable 或 Callable)封装成可定时调度的任务(RunnableScheduledFuture),然后将任务放入一个定制的延迟队列中。剩下的工作就都回归于 ThreadPoolExecutor 了。

在 ThreadPoolExecutor 中，如果队列有任务则拿出来直接执行即可，其中的队列只是普通的阻塞队列。而调度线程池中使用的是无界的延迟队列，并且在 poll 或 take 方法中，具有时间的判断，只有延时到期的任务才能从队列中出队并执行。

也正因为其中使用的是无界的延时队列，所以调度线程池没有 最大线程数 的控制，也就是说 ThreadPoolExecutor 中关于最大线程数的控制方法，对于调度线程池是无效的。

调度线程池具有以下特点：

  • 其内部任务全部转为定时可执行任务 ScheduledFutureTask，这个任务类是一个内部类，是专门定制的
  • 使用的延迟队列也是专门定制的，并且也是一个内部类 DelayedWorkQueue(同样基于堆实现)
  • 调度线程池中的任务具有执行顺序，也就是如果两个任务具有同样的延时时间，则按照 FIFO 的顺序执行
  • 没有最大线程数限制
  • 通过 ExecutorService 也可以提交任务，但被视为无延时任务，会被立即执行
  • 定时任务返回的 ScheduledFuture get 只会得到异常(任务取消或线程池结束)而没有结果，因为其重复执行逻辑使用的 FutureTask 的 runAndReset 方法(不设置结果，重置任务状态)。

调度线程池中，有三个重要的参数：

  • continueExistingPeriodicTasksAfterShutdown - 线程池关闭后，内部的调度任务是否继续执行
  • executeExistingDelayedTasksAfterShutdown - 线程池关闭后，内部的延迟任务是否继续执行
  • removeOnCancel - 任务取消后，是否立即从队列移除

其中有两个概念，PeriodicTask 与 DelayedTask。PeriodicTask 是指会被重复执行的任务，包括定时重复执行与延时重复执行。而 DelayedTask 是指非重复执行任务，但可能一开始执行时被设定了延时时间。这两个名字很容易与 scheduleWithFixedDelay 中的 delay 混淆。

Note ： 注意，只有 shutdownNow 方法给所有线程设置了中断标识，而其他的方法比如判断线程池是否可执行任务而取消任务时，mayInterruptIfRunning 都是 false 的，也就是不设置中断标识。并且，也只是取消就结束了，而不会拒绝任务

以上都是一些与周期任务执行的相关方法。以上比较重要的方法就是 run() 方法，其答题逻辑就是

  1. 如果当前线程池无法执行任务，则取消任务
  2. 如果任务属于非周期性任务，则调用普通的 run 方法执行即可
  3. 否则，调用 runAndReset 方法，执行并重置任务状态，重设任务下一次执行延时时间，并重新投入队列中

canRunInCurrentRunState 方法中，比较重要的是 periodic 判断。这里传进来的是一个 bool 值，true 表示周期性任务，false 表示非周期性任务。也就是说，在线程池终止后，周期性任务是否还能继续执行，由 continueExistingPeriodicTasksAfterShutdown 说了算，而非周期性任务是否还能执行，由 executeExistingDelayedTasksAfterShutdown 说了算。
