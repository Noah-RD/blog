Queue文档翻译

发表于 2021-03-24 | 分类于 并发

文本为对 JAVA SE 8 java.util 包下 Interface Queue<E> 文档的翻译。

接口概述

其数据结构被设计为一个，处理预先持有元素的集合。除了基础的 Collection 接口中定义的操作外，队列还提供额外的插入、提取、检索操作。所有的这些操作都存在两种形式，一种当操作失败抛出异常，另一种则返回特定值(有可能是 null 或者 false，取决于具体实现)。下面表格的插入操作是专门给有界队列所设计的。在大部分的实现中，插入操作不能失败。

队列通常但不一定以 FIFO(先进先出) 的方式对元素进行排序。例如优先级队列，其内部元素的顺序依赖一个比较器，或者元素的自然顺序。而栈则提供 LIFO(后进先出) 的元素顺序。无论哪种排序方式，当调用 remove() 或 poll() 方法都会移除队列的头元素。在 FIFO 队列中，所有的新元素都从队尾插入。其他类型的队列，可能会使用不同的放置规则。每个队列必须指定其排序属性。

offer() 方法如果可能的话，插入一个元素，否则返回 false。这与 Collection.add 方法不同，后者当插入元素失败时，只能抛出一个未经检查的异常。offer 方法被设计用来，当添加元素失败是正常的情况下，而不是一个异常操作。比如在有界队列中。

remove() 和 poll() 方法移除并返回队列的头元素。确切地说，从队列中删除了哪个元素是队列的排序策略而定的，每个实现的实现方法不同。remove() 和 poll() 方法唯一不同的地方在于队列为空时：remove 方法抛出异常，而 poll 方法返回 null。

element() 和 peek() 方法只是返回，而不会移除队列的头元素。

Queue 接口并没有定义关于阻塞队列的相关方法(通常在并发程序中使用)。这些方法一般是，当队列没有元素或队列元素已满时进行等待，这些都被单独定义在 BlockingQueue 接口中，继承自 Queue 接口。

队列的实现通常情况下，不允许插入 null 元素，但是也有一些实现，比如 LinkedList 并不限制插入 null 元素。但即使在允许插入 null 元素的实现中，也最好不要插入 null 元素，因为 null 被作为 poll() 方法的特殊返回值，以指示该队列不包含任何元素。

队列的实现通常不会定义基于元素的 equals 和 hashcode 方法，而是直接继承 Object 中的相关方法。因为对于具有相同元素但顺序属性不同的队列，基于元素的相等性并不总是很好定义的。

这个接口是 Java 集合框架一员。

方法概览

add

boolean add(E e)

如果可以立即将指定的元素插入此队列，而不会违反容量限制，则在成功时返回true，如果当前没有可用空间，则抛出 IllegalStateException。

offer

boolean offer(E e)

如果可以立即将指定的元素插入此队列，而不会违反容量限制。当使用容量受限的队列时，此方法通常比 add(E) 更可取，因为 add(E) 可能仅能通过引发异常来表示插入失败。

remove

E remove()

检索并删除此队列的头。此方法与 poll 的不同之处仅在于，如果此队列为空，则将引发异常。

poll

E poll()

检索并删除此队列的头部，如果此队列为空，则返回 null。

element

E element()

检索但不删除此队列的头。此方法与 peek 的不同之处仅在于，如果此队列为空，则将引发异常。

peek

E peek()

检索但不删除此队列的头部，如果此队列为空，则返回 null。
