ArrayBlockingQueue

发表于 2021-03-29 | 分类于 并发

ArrayBlockingQueue(有界阻塞队列) 源码学习。

概述

一个基于数组实现的有界阻塞队列。队列元素的顺序为 FIFO(先进先出)。队头元素是存在时间最长的，队尾元素是存在时间最短的。新元素从队尾插入，检索和获取元素等操作在对头进行。

这是一个典型的”有界缓存”，使用固定长度的数组来保存元素，生产者插入元素，消费者检索元素。当创建后，其队列容量则固定不可更改，在队列已满时插入元素，或队列为空时获取元素等操作会被阻塞。

此类支持可选的公平性策略，用于排序正在等待的生产者和使用者线程。默认情况下，不保证此顺序。但是，将公平性设置为 true 的队列将按 FIFO 顺序授予线程访问权限。公平通常会降低吞吐量，但会减少可变性并避免饥饿。

此类及其迭代器实现 Collection 和 Iterator 接口的所有可选方法。

此类是 Java 集合框架的一员。

源码详解

数据结构

ArrayBlockingQueue 其内部数据结构是数组，队列中的元素全部存储在数组 items 内。允许放入空元素。

其元素的插入与检索的操作，则基于双下标环形数组的算法。也就是当下标到达数组末尾时，则跳到开头继续。

ArrayBlockingQueue 是线程并发安全的，其控制并发的逻辑，就是经典的重入锁+双条件控制(一把全局锁)。

而在概述中，我们看到有可选的公平性策略，这个策略就是由重入锁提供的。

这部分知识属于 AQS 部分，简单说就是当锁资源释放时，所有等待锁的线程，在获取锁时的公平性。我们知道 AQS 中有个等待队列，如果是公平的，则按照队列中线程的顺序获取锁，否则线程是可以抢占锁的。公平下，线程顺序有保证，不会出现饥饿情况，但吞吐量低。而非公平下，吞吐量高，但会有线程饥饿的可能。

ArrayBlockingQueue 默认为非公平模式。

构造函数

ArrayBlockingQueue 有两种构造函数

两个参数，一个用于设置队列大小(有界),另一个用于设置线程访问公平性的，默认为 false(非公平)

第二种构造函数，多一个 Collection 参数，也就是使用集合 c 来初始化队列。当然，集合的长度不能超过 capacity，否则会报错。

队列操作

我们可以看到，ArrayBlockingQueue 是继承自 AbstractQueue 的。AbstractQueue 是一个抽象类，其中实现了三个最基本的操作，add、remove、element。

而这三个方法，都是基于 offer、poll、peek 的，区别就是抛出异常还是返回特定值。后面三个方法都是需要 AbstractQueue 的子类来实现的。

我们在看一下接口 BlockingQueue 中，对于这几个方法的一览表。

在看这些方法之前，我们先看一下这些方法中用到的三个基本操作，enqueue(进队)、dequeue(出队)、removeAt(删除指定位置的元素)

这是一个元素入队操作，操作很简单

  • 将待插入元素，放到下标 putIndex 处
  • 判断 putIndex 是否超出右边界，是则从数组头开始
  • 队列中元素数量计数 + 1
  • 释放锁，通知其他线程(这个由重入锁内部 AQS 来负责)

这是一个元素出队的操作，操作很简单

  • 拿出 takeIndex 下标处的元素
  • 判断 takeIndex 是否超出右边界，是则从数组头开始
  • 队列中元素数量计数 - 1
  • 通知同步迭代器，有元素更新操作
  • 释放锁，通知其他线程(这个由重入锁内部 AQS 来负责)

我们知道 ArrayBlockingQueue 最顶层是实现自 Collection 的，也就是 ArrayBlockingQueue 属于集合，而集合肯定是可以迭代的，也就是具有迭代器的。itrs.elementDequeued() 方法就是通知这个迭代器，其队列内部发生的变化，要求迭代器同步更新一下。

其迭代器具体实现，等有空了在分析源码(TODO)。

删除指定位置元素的操作，其实本质就是把待删除索引位置空。但为了保持入队和出队时的判空逻辑统一，所以删除操作使用了一个循环，来保证删除的位置 == 插入的位置。因为这两个位置的值都需要是 null。

接下里，我们看一下队列的操作都有哪些。

add 操作，内部就是调用父类的 add，其内部实现我们上面也说了，就是调用子类的 offer 方法。

offer 方法，元素入队成功则返回 true，如果队列已慢，则返回 false。使用重入锁，保证线程安全。

put 方法是一个阻塞方法。当队列满了无法插入元素时，则一直等待，直到 1. 插入成功 2. 被中断

这个 offer 也是阻塞版本，是 put 的具有等待时间的版本。其内部逻辑与 put 一样。

poll 出队操作。如果队列为空，则返回 null

take 为阻塞版的出队操作

这个 poll 也是阻塞版的出队操作，是 take 的带有时间的版本。

检索元素，其返回队列的头元素，但不移除。队列为空，则返回 null

其重写了 remove 方法。其删除的逻辑是，从 takeIndex 开始遍历数组到 putIndex 为止，当找到 o.equals(items[i]) 时，返回 true，表示找到了。否则返回 false。

注意，基于环形数组的删除操作，是非常慢且具有破坏性的操作，所以除非特殊情况下，否则不要随便用。其使用的理想情况是，仅在已知该队列无法被其他线程访问的情况下进行。

主要操作队列的方法，就是以上这些。最后还有几个方法如：contains、size、remainingCapacity(返回这个队列还能再放多少元素，当然是非阻塞的放)、toArray、clear等，源码都很简单，就不写了。
