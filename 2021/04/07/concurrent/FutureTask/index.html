FutureTask

发表于 2021-04-07 | 分类于 并发

FutureTask 源码学习。

简述

首先，上一张继承图

一个可取消的异步任务，其提供 Future 接口的基础实现，包括开始与取消任务，查看任务是否完成以及获取计算结果。只有当任务完成后才能获取计算结果，如果任务还没有结束，则 get 方法会被阻塞直到可以获取结果。一旦任务结束，这个任务则不能重新开始或取消，除非使用 runAndReset() 方法执行任务。

FutureTask 可以包装一个 Callable 或 Runnable 对象。FutureTask 也实现了 Runnable，所以 FutureTask 也可以直接提交给 Executor 执行。

除了作为独立类，FutureTask 提供了一些 protected 方法，用于创建自定义任务子类。

源码详解

首先，1.8 版本的 FutureTask 不在依赖 AQS 进行并发控制，主要是为了避免当并发取消任务时，所保留的中断状态，而使用户感到奇怪。1.8 版本的并发设计，使用一个 state 字段，通过 CAS 操作来跟踪任务的状态流转，并使用一个 Treiber 栈来保存需要等待的任务的线程。其内部为了提高性能，直接使用 unsafe 类。

接下里，就是最重要的状态流转。FutureTask 共有 7 个状态：

其状态流转共有四种可能：

  • NEW -> COMPLETING -> NORMAL
  • NEW -> COMPLETING -> EXCEPTIONAL
  • NEW -> CANCELLED
  • NEW -> INTERRUPTING -> INTERRUPTED

注意，CANCELLED 与 INTERRUPTING 状态，只能通过 NEW 来流转。

  1. 任务的初始状态为 NEW
  2. 而最终的 INTERRUPTED 状态只能通过 set、setException、cancel 方法流转。
  3. COMPLETING 状态只有在设置结果值时才会流转到，而在任务执行的时候，其实都是 NEW 的状态。这也是为什么，CANCELLED 与 INTERRUPTING 状态，只能通过 NEW 来流转。
  4. INTERRUPTING 只有在 cancel(true) 方法时，才会流转。true 表示是否中断。

其他的属性

构造方法

共有两个构造方法，逻辑都是设置 callable 任务，并设置 state 的初始状态为 NEW

操作方法

其内部，对于等待线程也就是阻塞在 get 上的线程，使用了一个 Treiber stack，这个 Treiber 其实就是一个链表，每当有一个新的等待线程结点创建后，直接放在链表的头部，就形成了一个栈(每次放在顶部)

run/runAndReset

首先，我们知道 FutureTask 也是一个 Runnable，所以其实现了 run 方法

我们可以看到，run 方法就是在运行任务，如果成功设置结果，否则异常则设置异常。这里可以看出来 run 任务的过程，state 状态一直是 NEW 的(如果没有其他线程修改的情况下)。我们看一下 set 与 setException 方法

可以看出来，COMPLETING 状态只要在 set 与 setException 时才会设置，并在设置完值/异常值后，都会有一个最终状态，NORMAL 与 EXCEPTIONAL。

finishCompletion() 方法用来通知所有，正在等待任务完成的线程(也就是阻塞在 get 方法上的线程)，并清空等待队列与 callable 任务。

与 run 很相似的一个方法是 runAndReset 方法，大部分逻辑是一样的，唯一不同就是正常完成任务后，不设置结果也就是不使用 set 方法，所以其 state 依然是 NEW。这个方法设计给多次执行的任务。

get

run 方法的运行是由异步线程执行的，用来执行任务。而与之相关的一个重要方法，就是 get。get 用于主线程获取任务执行的结果。

get 方法有两个版本，一个是无参数，一个支持超时。

get 方法内主要的就是 awaitDone 方法，用于阻塞当前线程等待任务完成。

可以看到 awaitDone 一共做了几件事

  1. 判断当前等待线程是否被中断
  2. 如果当前等待线程没有结点，则初始化一个
  3. 如果当前等待线程结点没有入队，则入队
  4. 调用 LockSupport 进行 park 操作阻塞等待

get(long timeout, TimeUnit unit) 操作是一样的，只不过多了一个超时判断

最后，get 方法的最后会调用一个 report 方法，用于返回结果或抛出异常

report 中涉及到一个 CANCELLED 状态，那这个状态怎么来的呢？接下来就是最后一个重要的方法，cancel

cancel

最开始，我们就说了 FutureTask 是一个可取消的任务，这个方法就是用来取消任务的。

cancel 方法非常简单，如果中断执行线程，则 state 先流转为 INTERRUPTING，接下来执行 thread.interrupt() ，等中断后 state 最终流转为 INTERRUPTED。
如果不中断取消，则状态流转为 CANCELLED。
最后执行等待线程的清理工作即可。

这里需要注意一点：

当 cancel(true) 去以中断的方式中断任务的执行时，除非在 Callable 的 call() 方法实现上设计成响应线程中断，否则是不会中断 callable.call() 方法的执行的，虽然不会中断任务的执行，但是不会设置 callable 的运行结果，在 get() 方法返回时抛出 CancellationException 异常。

因为 cancel 方法会设置 state 的状态为 CANCELLED 或 INTERRUPTED，而在 get 方法内会根据 state 选择返回结果还是抛出异常。所以只要取消任务就会抛出取消异常，而不管任务代码是否已经计算完成，也就是不会设置结果。

其他方法

注意 isDone 方法：对于完成的定义是，无论正常完成，还是抛出异常、还是被取消掉了，都算完成(使用的时候要注意)。

最后，FutureTask 的所有 CAS 直接使用 UNSAFE 魔法类。

总结

FutureTask 主要就是弄清几个状态的流转，而且是只有这几个流转状态。其源码就是根据这几个流转状态而写的。

NEW -> COMPLETING -> NORMAL
NEW -> COMPLETING -> EXCEPTIONAL
NEW -> CANCELLED
NEW -> INTERRUPTING -> INTERRUPTED
