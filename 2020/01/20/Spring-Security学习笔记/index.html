Spring Security学习笔记

发表于 2020-01-20 | 分类于 学习笔记

Spring Security

Interface AuthenticationManager

处理一个 Authentication 请求。

尝试对传递的 Authentication 对象进行身份验证，如果成功，则返回完全填充的 Authentication 对象（包括授予的权限）。

AuthenticationManager 必须遵守以下有关异常的约束：

  • 如果禁用了帐户，并且 AuthenticationManager 可以测试此状态，则必须抛出 DisabledException。
  • 如果帐户被锁定，并且 AuthenticationManager 可以测试帐户锁定，则必须抛出 LockedException。
  • 如果提供的凭据不正确，则必须引发 BadCredentialsException。 尽管上述例外是可选的，但 AuthenticationManager 必须始终测试凭据。

应按上述顺序对异常进行测试，并在适用的情况下按上述顺序进行抛出（即，如果帐户被禁用或锁定，则立即拒绝身份验证请求，并且不执行凭据测试过程）。 这样可以防止针对禁用或锁定的帐户测试凭据。

Class ProviderManager

通过 AuthenticationProviders 列表迭代 Authentication 请求。

AuthenticationProviders 通常按顺序尝试，直到提供非空响应为止。非空响应表示提供者有权决定认证请求，并且不再尝试其他提供者。如果后续提供者成功验证了请求，则将忽略先前的验证异常，并使用成功的验证。如果没有后续提供者提供非空响应或新的 AuthenticationException，则将使用最后收到的 AuthenticationException。如果没有提供者返回非 null 响应或可以处理 Authentication，则 ProviderManager 将抛出 ProviderNotFoundException。还可以设置父 AuthenticationManager 并尝试验证，如果没有配置的 providers 可以执行身份验证。尽管此功能旨在支持名称空间配置选项，但通常不是必需的功能。

此过程的 exception 是 provider 抛出 AccountStatusException 时，在这种情况下，将不再查询列表中的其他 providers。身份验证后，如果 Authentication 对象实现了 CredentialsContainer 接口，则将从返回的 Authentication 对象中清除 credentials。可以通过修改 EraseCredentialsAfterAuthentication 属性来控制此行为。

事件发布

身份验证事件发布被委派给已配置的 AuthenticationEventPublisher，默认情况下，该事件为不执行事件的空实现，因此，如果您自己配置 Bean，则要接收事件就必须注入 publisher bean 。标准实现是 DefaultAuthenticationEventPublisher，它将常见的异常映射到事件（在身份验证失败的情况下），如果身份验证成功，则发布 AuthenticationSuccessEvent。如果使用名称空间，则 <http> 配置将自动使用此 bean 的实例，因此您将自动从应用程序的 Web 部件接收事件。

请注意，如果实现已从“父” AuthenticationManager 获得认证结果（或异常），则该实现还会发布认证失败事件。因此，在这种情况下，通常不应将父级配置为发布事件，否则将存在重复项。
