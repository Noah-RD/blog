基于docker搭建redis一主两从三哨兵

发表于 2020-02-29 | 分类于 运维

Sentinel(哨兵)是主从模式的升级版，在主从模式下，一旦主节点由于故障不能提供服务，需要手动将从节点晋升为主节点，同时还要通知客户端更新主节点地址，这种故障处理方式从一定程度上是无法接受的。于是便有了具有自动故障转移与恢复的哨兵功能。

简介

Sentinel 的工作原理网上或官网都有很多介绍，简单说，就是哨兵集群监控 redis 主从结点，如果某个哨兵监控到主节点故障，会将此主节点标记为 s_down(主观下线),然后与其他所有哨兵开始对主节点投票。
如果有足够的哨兵同意主节点不可用，便将该主节点标记为o_down(客观下线)，然后从其他从结点中选取一个结点升为主节点，并将其他从结点指向该升级后的主节点。

搭建

由于哨兵采用投票的方式，所以至少需要三个哨兵结点，并且应该放到不同的机器上，甚至不同的物理区域。官网上有针对几种最小部署的架构图，可以进行参考。中文文档

本篇文章分为两个搭建部分，一个是本地环境测试，也就所有结点跑在一台机器上，然后是服务器多台环境搭建。本文使用 docker 容器，并用 docker-compose 编排文件搭建部署。

这里有个坑需要注意下，这个配置文件不是只读的，也就是哨兵会在启动后、故障转移等时写入一些配置，这可能导致一些问题。比如在我测试过程中，发现配置的一些指令丢失，或者由于版本问题，导致配置文件内容行数不一样时，会覆盖或丢失一些配置。
所以这里建议，将源配置文件所有注解去掉，只保留指令部分，这样也容易排查与查看指令部分，而不用翻来翻去的。

然后我们就可以测试了，比如挡掉主节点，测试是否会自动故障转移，测试方法网上很多，就不赘述了。这里给一个测试的命令，不需要停止主节点去测试，只需要进入容器执行redis 命令即可模拟。

关于读写分离，读负载均衡这部分，是客户端框架支持的，比如 java 的 lettuce 或 redisson 等。
然后这个哨兵部署的高可用 redis 就可以使用了，但是当用应用连接的时候有个坑，比如 java 的 redis 客户端 redisson。

redisson 会在注册哨兵的时候，对哨兵使用 SENTINEL SENTINELS MYMASTER 命令，获取其他哨兵的信息，比如 ip、port，然后会去测试这些哨兵是否可用。
但是由于我们使用容器搭建，哨兵命令返回的是容器内部的 ip，而并非宿主机的 ip，所以肯定是无法连接的。所以，我们需要将 java 应用也放到容器中，通过容器自动获取 ip 的方式，即可使用。
当然，我们也可以通过修改容器网络模式为 host 模式，不做网桥转发，而是直接用本地 ip。但是这样安全性很低，而且你的 redis 主从都需要映射端口到宿主机才行。
