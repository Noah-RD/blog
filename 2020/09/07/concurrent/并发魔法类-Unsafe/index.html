并发魔法类-Unsafe

发表于 2020-09-07 | 分类于 并发

一个神奇的 JUC 工具类 ^ ^

导读

可能很多人没听说过 Unsafe 这个类，首先这个类不是JDK 的标准，它是Sun 的内部实现，存在于sun.misc包下，Oracle 发行的JDK中不包含其源码。那我们为什么要学习这个类呢？如果你看过 JUC 并发包源码，你会发现几乎在底层都使用到了这个类。那这个类干什么用的呢？

  1. 可以绕过JVM 直接操作内存，包括内存的申请、修改、复制、释放、访问
  2. 提供管程模型的获取锁与释放锁指令操作
  3. 提供 JDK 层面的锁实现 park/unpark
  4. 提供 CAS 指令操作
  5. 可以绕过 JVM 的安全检查去动态创建和初始化类
  6. 提供内存屏障的指令操作

可以说 JUC 在底层系统层面的支持，就是依靠这个类，想要学好 JUC，可以说这个类是基础。但在我们应用代码中不会直接用到这个类，出于安全考虑，这个类通常只能由标准类库使用，它在提升java 运行效率、增强java 语言底层操作能力起着很大作用，一些高性能类库如Netty、Kafka等都有使用。

本文使用 OpenJDK8中的 Unsafe 源码，由于 Unsafe的实现是和虚拟机实现相关的，所以不同的虚拟机实现它们的对象结构可能不一样，这个 Unsafe只适用于Hotspot虚拟机。

Unsafe 实例

Unsafe为调用者提供了非安全的操作能力，由于 Unsafe 提供了非常底层的操作，可以读写任意内存地址的数据，所以我们千万不要把 Unsafe 暴露到生产环境或传递给不受信任的代码库中。

可以看到，Unsafe 的构造方法是私有的，构造的 theUnsafe 是单例的且只能通过 getUnsafe()方法拿到，而 getUnsafe() 方法注解为 @CallerSensitive 的，也就是对调用者是敏感的。Unsafe 默认只能由系统类加载器(Bootstrap ClassLoader)加载，否则会抛出Security 安全异常。

我们可以看一下 VM.isSystemDomainLoader 是如何判断的

判断是否为系统类加载器加载很简单，只要加载器 == null 即可。我们知道Bootstrap ClassLoader是由 JVM 实现的，在应用代码中是获取不到引用的，也就是 null。

直接通过内存拿到值

Unsafe 内，有一组这样的操作，可以通过内存地址，直接拿到对象，或通过对象的内存地址，拿到对象字段的值或引用。这里首先说一下，Unsafe 虽然可以直接操作内存，但也是 JVM 管理内的内存，否则会使 JVM 进程退出(我们知道操作系统不同进程之间是不允许直接访问内存资源的)。

获取指定对象中指定偏移量的字段或数组元素的值。其中 o 是 java 堆对象，offset 是要访问字段或数组元素的偏移量。如果参数 o 为 null，则获取内存中该地址的值(offset 变为内存绝对地址)

与 get 方法大致相同，put则是将值存储到 java变量中。

在深入解释这两个方法前，我们需要先看一下如何获取偏移量

这几个方法分别是获取静态字段、非静态字段以及数组字段的偏移量信息。

  1. objectFieldOffset。获取某字段在对象中的内存偏移量。
    . 该方法为了和其他方法保持一致，所以使用long 返回类型(其实不需要那么多比特位来编码非数组对象的偏移量，毕竟一个字节就能表示 256个成员变量。这里用 long 表示了 8 个字节)。
  2. staticFieldBase。获取该字段是属于哪个对象的。和 staticFieldOffset 一起使用。
  3. staticFieldOffset。获取某静态域字段在对象中的内存偏移量。
    1. 对于任何给定的字段，该方法总是返回相同的值；同一个类的不同字段总是返回不同的值。
    2. 从 1.4.1 开始，字段的偏移以 long 表示，虽然 Sun 的 JVM 只使用了 32 位，但是那些将静态字段存储到绝对地址的 JVM 实现需要使用 long 类型的偏移量，通过 getXX(null,long) 获取字段值，为了保持代码迁移到 64 位平台上 JVM 的优良性，必须保持静态字段偏移量的所有比特位
  4. arrayBaseOffset。获取数组第一个元素在内存中的基础偏移量。
  5. arrayIndexScale。获取数组中每个元素在内存中所占的字节。

arrayBaseOffset 与 arrayIndexScale，也用于初始化 Unsafe 的几个静态变量

我们继续回到 getInt() 这个方法。

其内存访问需要符合以下条件之一，否则该方法的返回值是不确定的

  1. offset 通过objectFieldOffset方法获取的类的某一字段的偏移量，并且关联对象o必须是该类的兼容对象(对象 o所在的类或该类的子类)
  2. offset 通过 staticFieldOffset 方法获取的类的某一静态字段偏移量，o 是通过 staticFieldBase 方法获取的对象
  3. 如果 o 引用的是数组，则offset 的值为 B + N * S，其中 B 是通过 arrayBaseOffset获取的数组第一元素在内存中的基础偏移量，N是数组的合法下标，S 是通过 arrayIndexScale方法获取的数组元素所占字节数。
  4. 要访问的字段类型和方法的返回类型要相同

符合以上任意条件就可以正确访问 java 对象的字段。这个方法使用两个参数引用一个变量，我们称其为java变量提供了 double-register 模型。如果参数 o 为 null，offset 变为内存绝对地址，我们称其为非 java 变量提供single-register 模型。

这里千万注意，double-register的 offset 是偏移量，而single-register的 offset 是绝对地址，千万不要搞混。源码中 double-register 和 single-register 方法都很相似，大致如下

所谓小端模式是指：字节数据的低位存储在内存的低字节，高位则存储在内存的高字节。大端模式则正好相反：字节数据的高位存储在内存的低字节，而低位存储在内存的高字节。

内存空间操作

Unsafe涉及内存的分配操作，主要有以下几个方法

  1. allocateMemory：分配指定大小的一块本地内存，分配的这块内存不会初始化，其内容通常是没用的数据(原注释中是garbage)。返回的本地指针不会是 0，且该内存是连续的。调用freeMemory可以释放此内存，调用reallocateMemory可以重新分配
  2. reallocateMemory：重新分配一块指定大小的本地内存，超出老内存块的字节不会被初始化，其内容通常是没用的数据。当且仅当请求的大小为 0 时，此方法返回本地指针会是 0。该内存块是连续的，调用freeMemory可以释放此内存，调用reallocateMemory可以重新分配。参数 address可以是 null，这种情况和allocateMemory是一样的效果
  3. setMemory：将给定的内存块的所有字节设置成固定的值(通常是 0)。该方法通过两个参数确定内存块的基准地址，因此提供了double-register 模型。如果引用的对象是 null，则 offset会被当做绝对基准地址。该写入操作是按单元写入的，单元的字节大小由地址和长度参数决定。每个单元的写入是原子性的
    1. 如果地址和长度都是8 的倍数，则一个单元为 long 型
    2. 如果地址和长度都是 4 的倍数，则一个单元为 int 型
    3. 如果地址和长度都是 2 的倍数，则一个单元为 short 型
  4. copyMemory：复制指定内存块的字节到另一块内存
  5. freeMemory：释放通过allocateMemory 或reallocateMemory获取的内存，如果参数 address 为 null，则不作任何处理

allocateMemory、reallocateMemory、setMemory、freeMemory分别是对C函数malloc、realloc、free、memset的封装，这样 Unsafe 就提供了动态获取和释放本地方法内存的功能。

  1. malloc：用于分配一块全新的未使用的连续内存，但该内存不会初始化，即不会被清零。
  2. realloc：用于内存的缩容和扩容，有两个参数：从 malloc返回的地址和要调整的大小。该函数和 malloc 一样，不会初始化，它能保留之前放到内存里的值，很适合做扩容。
  3. free：用于释放内存，该方法只有一个地址参数，那它如何知道要释放多少个字节呢？其实在 malloc分配内存的时候会多分配4 个字节用于存放该块的长度。比如 malloc(10)其实会分配14 个字节，这样 free函数只要读取前 4个字节就知道要释放多少内存了。
  4. memset：一般用于初始化内存，可以设置初始化内存的值，一般初始化会设置成0.即清零操作。

park/unpark

将一个线程进行挂起是通过 park 方法实现，调用 park 后，线程将一直阻塞直到超时或者中断等条件出现。unpark 可以终止一个挂起的线程，使其恢复正常。整个并发框架中对线程的挂起操作被封装在 LockSupport 类中，LockSupport 类中有各种版本 pack 方法，但最终都调用的 Unsafe.park() 方法。

unpark 函数为线程提供 “ 许可（permit）”，线程调用 park 函数则等待 “ 许可 “。这个有点像信号量，但是不同于信号量，这个许可是不能叠加的。在一个线程上，无论连续调用多少次 unpark，也只会颁发一个“许可”，之后执行 park 使用了这个“许可”后，再次执行 park 线程依然会被阻塞。

unpark 函数可以先于 park 调用（但最好别这样做），比如线程 B 调用 unpark 函数，给线程 A 发了一个 “ 许可 “，那么当线程 A 调用 park 时，发现已经有 “ 许可 “，会马上再继续运行。park 遇到线程终止时，会直接返回（不同于 Thread.sleep，Thread.sleep 遇到 thread.interrupt() 会抛异常）。unpark 无法恢复处于 sleep 中的线程，只能与 park 配对使用，因为 unpark 发放的许可只有 park 能监听到。

CAS 操作

Unsafe 不仅提供了 CAS 的常用操作，还可以把非volatile语义的变量执行volatile操作。

这是具有 volatile 语意的 get 和 put方法。volatile 语意为保证不同线程之间的可见行，即一个线程修改一个变量之后，保证另一线程能观测到此修改。这些方法可以使非 volatile 变量具有 volatile 语意。

有三类很相近的方法：putXx、putXxVolatile 与 putOrderedXx

  1. putXx 只是写本线程缓存，不会将其它线程缓存置为失效，所以不能保证其它线程一定看到此次修改
  2. putXxVolatile 相反，它可以保证其它线程一定看到此次修改
  3. putOrderedXx 也不保证其它线程一定看到此次修改，但和 putXx 又有区别，它的注释上有两个关键字：顺序性（Ordered）和延迟性（lazy），顺序性是指不会发生重排序，延迟性是指其它线程不会立即看到此次修改，只有当调用 putXxVolatile 使才能看到

Unsafe 中还有一些其他方法，比如统计 CPU 负载、获取本地内存指针、创建匿名类(这个匿名类是虚拟机级别的，和我们写的匿名内部类不是一个概念。有兴趣可以看这里)等。但是大部分并发中常用的方法已经介绍完了。
